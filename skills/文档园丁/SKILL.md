# 文档园丁

知识库建立后的**唯一维护入口**。核心职责：让文档保持精简、准确、与代码一致；主动从代码中提取未记录的决策和业务规则。

## 核心原则

1. **绝对不生成任何临时性文档**（变更日志、巡检报告、审计报告、测试报告、完成报告）。发现问题直接修复，修复结果在对话中口头汇报即可。
2. 可复用的经验和规则写入 Cursor rules（`.cursor/rules/`），不写独立 .md 文件。
3. **权威等级优先**：处置任何文档前，先读 frontmatter 的 `authority` 字段，按等级决定处置权限。

## 文档权威等级感知

文档头部的 `authority` frontmatter 决定园丁的操作边界：

| 等级 | 含义 | 园丁权限 |
|------|------|----------|
| **L1** | 商业决策 (WHY) | 绝不删除、绝不合并。内容更新需人类确认。只能修正格式错误。 |
| **L2** | 判定逻辑 (WHAT) | 可建议修改，必须展示 diff 并获得人类确认后执行。 |
| **L3** | 系统机制 (HOW) | 可自行更新以与代码保持一致。 |
| 无标记 | 未分级 | 整理时根据内容特征判定层级，主动补充 frontmatter，向用户确认。 |

**关键规则**："多份文档说同一件事"仅适用于**同一权威等级**的文档。不同 authority 层级的文档即使覆盖同一业务领域，也不算重复——它们服务于不同读者、有不同的修改权限。

## 作用域感知

扫描脚本会自动检测目标目录的类型：

| 模式 | 自动判断条件 | 巡检范围 |
|------|------------|---------|
| **项目级** | 目标目录有 `.git` | 检查项目内部文档一致性 |
| **服务器级** | 目标目录无 `.git` 且有 ≥2 个独立项目 | 检查服务器文档 + 子项目索引 |

## 子 agent 使用规范

所有涉及多文件、多模块的操作，必须使用 Task 工具启动子 agent 并行处理（每批最多 4 个）。

## 能力A：巡检

**触发**："文档巡检" / "跑一下文档园丁" / "检查文档"

### 流程

1. 运行扫描脚本：`bash ~/.cursor/skills/文档园丁/scripts/scan.sh <项目根目录>`
2. 根据扫描结果，启动并行子 agent 分头直接修复（**处置前先检查目标文档的 `authority` 字段**）
3. 在对话中汇总修复了什么，**不生成任何报告文件**

## 能力B：整理散乱文档

**触发**："整理文档" / "整理XX模块的文档"

### 处置标准

**前置步骤**：对每份文档先检查 frontmatter `authority` 字段，确定权威等级后再处置。

| 判断 | 动作 | 权威等级限制 |
|------|------|-------------|
| 仍在指导开发的架构/规范 | 迁移到 `文档/` 对应子目录 | 迁移时保留原 authority 标记 |
| 人类审核过的业务决策 | 保留并迁移，标记 L1/L2 | 不可删除或合并 |
| 多份**同等级**文档说同一件事 | 合并为一份 | **仅限同一 authority 等级** |
| 已落地的设计方案 | 删除（代码即文档） | 仅限 L3 或无标记文档 |
| 一次性分析/报告 | 删除 | 仅限无 authority 标记的文档 |

## 能力C：审计模块旧文档

**触发**："审计XX模块的文档" / "梳理XX模块"

1. 定位模块内所有 `.md` 文件
2. 子 agent 并行验证每份文档与代码的一致性
3. 识别人类决策内容（必须保留）vs 实现细节（代码已包含，可删除）
4. 扫描代码中未被文档记录的业务规则和设计取舍，发现后按能力D流程处理
5. 对已有文档执行处置（按权威等级规则），口头汇报结果

## 能力D：决策提取

**触发**："提取决策" / "记录决策" / 审计模块时自动执行

### 提取范围

| 类型 | 代码中的信号 | 记录到 |
|------|------------|--------|
| 商业规则 | 佣金比例、分配逻辑、审批条件、金额阈值 | `产品规格/` (L1/L2) |
| 设计取舍 | 注释中的"之所以""因为"、git 历史中的选型讨论 | `设计文档/` (L2) |
| 判定规则 | 状态机、枚举分类、校验条件 | `产品规格/` (L2) |
| 架构约束 | 性能边界、平台限制、不可变更的技术依赖 | `设计文档/` (L3) |

### 权限

- 提取结果始终先展示给用户，**不直接写入 L1/L2 文档**
- L3 级的技术约束可直接写入 `设计文档/`

## 自动修复权限

| 操作 | 权限 |
|------|------|
| 更新架构.md / AGENTS.md | 直接修复 |
| 更新/删除 L3 文档 | 直接修复 |
| 删除过期文档/空目录（无 authority 标记） | 直接修复 |
| 修改 L2 文档内容 | 展示 diff，人类确认后执行 |
| 修改/删除 L1 文档 | **禁止**。仅可修正格式错误 |
| 为无标记文档补充 authority frontmatter | 向用户确认 |
| 能力D提取的 L1/L2 级决策 | 展示草稿，用户确认后写入 |

# 项目结构化文档库构建

> **本技能仅用于首次搭建**。文档库建好后的所有维护统一由文档园丁技能负责。

## 核心理念

**给 Agent 一张地图，而不是一本说明书。**

1. **渐进式披露**：`AGENTS.md` 是 Cursor 原生 AI 入口文件，指向 `文档/` 深层文档
2. **不在仓库里 = 不存在**：把隐性知识搬入仓库
3. **为 Agent 可读性优化**：文档不含代码示例，充分信任大模型
4. **只记录长期有效的知识**：禁止创建变更日志、巡检报告、审计报告等临时性文档

## 作用域

本技能支持两种模式，根据目标目录自动判断：

| 模式 | 判断条件 | AGENTS.md 定位 | 架构.md 定位 |
|------|---------|---------------|-------------|
| **项目级**（默认） | 目标目录有 `.git`（即使子目录有多个 docker-compose） | 项目入口（≤250行） | 代码结构 + 部署拓扑 |
| **服务器级** | 目标目录无 `.git` 且子目录中有 ≥2 个含 docker-compose 或 AGENTS.md 的独立项目 | 服务器地图（≤150行） | 基础设施总览 |

### 服务器级与项目级的差异

| 维度 | 服务器级 | 项目级 |
|------|---------|--------|
| AGENTS.md 内容 | 项目索引表、服务器角色、关键约束 | 技术栈、模块结构、编码规范 |
| 架构.md 内容 | Nginx、SSL、域名、共享数据库、容器清单 | 业务域、代码分层、模块依赖 |
| `文档/` 子目录 | 仅 `凭据与配置/`、`操作手册/` | 完整目录（核心信念、设计文档、产品规格等） |
| 对子项目 | 只索引，不深入内部 | 不涉及 |
| git 同步项目 | 标注"文档面向生产"，不修改其内部文档 | 正常管理 |

## 命名规范

**一切使用中文命名**：文件、目录、导航文档、架构文档全部中文，`.md` 格式。

## 项目级目录结构

```
项目根目录/
├── AGENTS.md              # Cursor 原生 AI 入口文件（≤250行）
├── 架构.md                # 项目鸟瞰图
├── 文档/                  # 结构化知识库（唯一真相源）
│   ├── 核心信念/          # 团队工程原则与长期规范
│   ├── 设计文档/          # 仍在指导开发的架构设计
│   ├── 产品规格/          # 业务规则、判断逻辑、决策标准
│   ├── 参考资料/          # API文档、第三方对接、数据库Schema
│   ├── 凭据与配置/        # 开发阶段直接记录明文
│   ├── 操作手册/          # 操作SOP
│   └── 执行计划/
│       ├── 进行中/
│       └── 技术债务/
```

**注意**：不创建 `变更日志/`、`测试报告/`、`质量评分/`、`执行计划/已完成/` 目录。

## 服务器级目录结构

```
服务器根目录/
├── AGENTS.md              # 服务器地图（≤150行）
├── 架构.md                # 基础设施总览
├── 文档/                  # 服务器级共享文档（精简）
│   └── 凭据与配置/        # 跨项目共享凭据
├── 项目A/                 # 子项目（各自有 AGENTS.md）
├── 项目B/                 # git 同步的项目（文档面向生产，不触碰）
└── scripts/               # 运维脚本
```

## AGENTS.md 编写规则

### 项目级

Cursor 原生文件，AI 助手启动时自动读取，是 AI 理解项目的**唯一入口**。

**严格 ≤250 行**。必含：项目概述、技术栈、知识库索引、目录结构、标准模块结构、核心编码规范、部署约束、凭据位置。
**禁止放入**：详细 API 文档、完整部署步骤、数据库 Schema、代码示例。

### 服务器级

**严格 ≤150 行**。必含：服务器角色、关键约束、项目索引表（项目名/路径/域名/技术栈/是否 git 同步）、知识库索引、运维脚本。
**禁止放入**：任何项目的内部细节（模块结构、编码规范）。
对 git 同步的项目须标注"文档面向生产环境"。

## 架构.md 编写规则

### 项目级

必含：业务域划分、代码包分层、模块依赖关系、部署拓扑（容器/端口/域名）、数据存储。

### 服务器级

必含：域名体系（生产/测试/独立项目三类域名映射）、SSL 证书、Nginx 反向代理、全部容器清单（按项目分组）、共享基础设施、crontab、与生产服务器的差异对比。

## 文档分类规则

| 类型 | 说明 | 何时更新 |
|---|---|---|
| 核心信念 | 不随任务变化的长期约定 | 规则变化时 |
| 设计文档 | 仅保留仍在指导开发的架构设计，落地完成的删除 | 架构变化时 |
| 产品规格 | 业务规则、判断标准、人类审核过的决策 | 决策变化时必须同步更新 |
| 执行计划 | 进行中的任务；完成后直接删除，不归档 | 任务推进时 |
| 凭据与配置 | Token/密码/API Key 集中明文记录 | 配置变化时 |

## 初始化工作流（使用子 agent 并行）

### 项目级工作流

#### 阶段1：搭建结构 + 扫描现有文档（并行）

| 子 agent | 任务 |
|---------|------|
| agent-A | 创建 `文档/` 完整目录结构 |
| agent-B | 递归扫描项目下所有 `.md` 文件，生成文档清单 |

#### 阶段2：编写核心文档 + 分类现有文档（并行）

| 子 agent | 任务 |
|---------|------|
| agent-C | 编写 `架构.md`（读取代码结构、docker-compose、nginx配置） |
| agent-D | 编写 `AGENTS.md`（基于架构.md和文档结构） |
| agent-E | 将系统级文档迁移到 `文档/` 对应子目录 |
| agent-F | 整合凭据信息（读取 .env，生成凭据总览） |

#### 阶段3：收尾

1. 清理空目录
2. 口头汇报初始化结果（不生成报告文件）

### 服务器级工作流

#### 阶段1：扫描服务器状态（并行）

| 子 agent | 任务 |
|---------|------|
| agent-A | 扫描所有容器（docker ps）、Nginx 配置、SSL 证书、crontab |
| agent-B | 扫描各子项目目录（技术栈、docker-compose、是否有 AGENTS.md） |

#### 阶段2：生成文档（并行）

| 子 agent | 任务 |
|---------|------|
| agent-C | 编写 `架构.md`（基于阶段1收集的基础设施信息） |
| agent-D | 编写 `AGENTS.md`（项目索引 + 知识库索引） |
| agent-E | 创建 `文档/凭据与配置/`，整合跨项目共享凭据 |

**关键约束**：不修改 git 同步子项目的任何内部文档。

#### 阶段3：收尾

1. 口头汇报初始化结果（不生成报告文件）

# /执行测试用例

根据项目中的测试用例文件，对指定模块执行冒烟或回归测试。

## 调用方式

用户说 `/执行测试用例`、`跑冒烟`、`测试XX模块` 时触发。

## 输入解析

从用户消息中识别两个参数：
- **范围**：具体模块名或 `全部`
- **层级**：`冒烟`（仅标记为冒烟的核心用例）或 `回归`（模块全量）。默认冒烟。

## 执行流程

### 第一步：准备（主 agent 执行）

1. 扫描项目中的测试用例目录（常见位置：`文档/测试用例/`、`tests/`、`docs/test-cases/`）
2. 读取覆盖矩阵或用例索引，确定待测模块和冒烟用例编号
3. 读取各模块测试用例文件，提取标记为「冒烟」（冒烟层级）或全部用例（回归层级）
4. 准备认证 Token（按项目认证规范签发测试 Token）
5. 确认各服务运行状态（docker ps 或进程检查）

### 第二步：并行 API 测试（多子 agent 同时执行）

为每个待测模块启动一个 **shell 类型子 agent**（subagent_type="shell"），各 agent 独立工作互不干扰。

#### 子 agent 任务要求

每个子 agent 接收模块名、认证 Token、服务端口、待执行用例列表（含编号、步骤、期望），按以下规则执行：

1. 先检查健康端点，失败则报告「阻塞：服务未启动」并终止
2. 逐条执行用例：
   - **API 类用例**：构造 curl 请求，验证响应状态码和 body 结构
   - **配置类用例**：GET 配置端点，验证返回结构合理
   - **权限类用例**：分别用有权限和无权限 Token 请求，验证拦截行为
   - **前端路由类用例**：curl 前端端口对应路径，验证 HTTP 200
3. 每条用例报告：编号、结果（通过/失败/阻塞）、失败原因
4. 最后汇总：通过数/失败数/阻塞数

### 第三步：UI 截图验证（顺序执行）

API 测试全部完成后，对前端页面做快速截图验证（需要 Playwright MCP 或无头浏览器工具）。

对每个有前端的模块：
1. 导航到模块前端首页，注入认证 Token
2. 依次访问冒烟路由列表中的每条路由
3. 每个路由等待页面加载完成后截图
4. 检查页面是否白屏（body 文本长度过短视为异常）

每个模块在一次调用内完成所有路由截图，避免多 agent 抢浏览器。

### 第四步：汇总（主 agent 执行）

收集所有子 agent 返回的结果，输出汇总表格：
- 各模块通过/失败/阻塞数
- 失败用例明细（逐条列出原因）
- 截图异常（白屏或加载失败的页面）

## 并行安全保障

- **API 测试**：每个子 agent 使用 Shell 工具执行 curl，各 agent 独立进程，端口不同，零冲突
- **UI 截图**：在 API 测试全部完成后顺序执行，不存在多 agent 抢浏览器
- **认证隔离**：Token 统一生成一次传递给所有 agent，不依赖浏览器 Cookie/Session

## 注意事项

- 需要第三方客户端环境的用例，API 测试中跳过并标记「阻塞：需第三方环境」
- 纯小程序前端仅测后端 API
- 截图不替代人工验证——截图确认页面能渲染，业务正确性以 API 测试结果为准
